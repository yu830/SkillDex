# SkillDex Content Guidelines

SkillDex should be useful as a portfolio surface without overstating project maturity.

## Claims

- Describe what exists now, not what is planned.
- Do not claim deployment, users, stars, production adoption, certifications, or benchmarks without evidence.
- Use `published` only when a real public repo, demo, docs, or case study link exists.
- Use `local` for work that exists locally or in private review but is not publicly linked.
- Use `draft` for concept-stage work.
- Leave unknown links blank; the UI will show them as `TBD`.

## Skill Cards

Good skill summaries explain:

- what workflow the skill supports;
- which tool surfaces it fits;
- what risk or safety boundary matters.

Avoid vague claims such as "best", "production-grade", or "fully automated" unless there is proof.

## Project Cards

Project cards should state:

- project purpose;
- current status;
- core technical tags;
- real repo, demo, docs, or case study links when available;
- evidence notes that separate verified state from missing public proof;
- 2-3 concrete highlights grounded in the project scope.

If a project is not yet public, keep links blank rather than inventing placeholders.

Do not mark a project as `published` unless at least one real public link is present.

When evidence exists, prefer auditable references such as a repository path, commit, screenshot, validation log, or public URL. If those references are not available, write the missing item as `TBD`.
